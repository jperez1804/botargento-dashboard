## 1. Project Overview

### Vision

`botargento-dashboard` is a client-facing analytics dashboard for BotArgento tenants. Each WhatsApp-automation client (starting with real-estate brokerages) gets their own white-labeled dashboard at `dashboard.<clientN>.botargento.com.ar` where their team can monitor inbound volume, lead qualification flow, handoff activity, and follow-up priorities in real time.

The dashboard is BotArgento's sales card: a visible, polished artifact that demonstrates the value of the WhatsApp automation to prospects and keeps existing clients engaged. It is read-only in v1 — no write actions beyond CSV export.

Architecturally, it is the fourth pillar of the BotArgento stack, sitting alongside:
- `botargento-landingpage` (public marketing site)
- `botargento-embedded-signup-backend` (Railway control plane)
- `whatsapp-automation-claude` (per-tenant n8n runtime + Postgres)

The dashboard reads directly from the tenant's existing Postgres reporting views — no duplication of the reporting pipeline, no Google Sheets dependency, no 10-minute staleness lag.

### Goals

- Ship a client-facing reporting UI that makes WhatsApp automation value self-evident.
- Reuse the existing Postgres reporting views (`v_daily_metrics`, `v_flow_breakdown`, `v_contact_summary`, `v_handoff_summary`, `v_follow_up_queue`) with zero duplication.
- Deploy per tenant via Docker + Traefik, matching the existing n8n operational model.
- Build a vertical abstraction layer so real-estate is v1 but retail, services, and future verticals are a ~1-hour config-file addition.
- Ship full CI/CD with safe rollback from day one.

### Success Metrics

- **Time-to-deploy a new tenant dashboard:** ≤ 15 minutes of manual work.
- **Time-to-add a new vertical:** ≤ 1 hour (config file + n8n intent key verification).
- **First-paint data latency:** ≤ 500 ms (server-rendered, no client-side fetch round-trip on initial load).
- **Dashboard downtime per deploy update:** ≤ 5 seconds per tenant (rolling Docker restart).

## 4. Data Model

### Read surface (existing — provided by tenant Postgres)

Schema: `automation` (already exists on every tenant's DB, defined in `whatsapp-automation-claude/postgres-setup.sql`). The dashboard **reads only**, never writes.

**Views consumed by the dashboard:**

| View | Shape | Used by |
|------|-------|---------|
| `automation.v_daily_metrics` | One row per day (BA tz): `day`, `inbound_count`, `outbound_count`, `unique_contacts`, `handoff_count`, `handoff_rate` | Overview KPIs, 7-day volume chart |
| `automation.v_flow_breakdown` | One row per (day × intent × route): `day`, `intent`, `route`, `inbound_count` | Compatibility check; chart now dedupes contacts from `lead_log` |
| `automation.v_contact_summary` | One row per contact: `contact_wa_id`, `display_name`, `first_seen`, `last_seen`, `message_count`, `last_intent`, `handoff_count` | Conversations list, Top Contacts |
| `automation.v_handoff_summary` | One row per handoff target: `target`, `count_all_time`, `count_24h` | Handoffs summary card |
| `automation.v_follow_up_queue` | Prioritized contacts: `contact_wa_id`, `display_name`, `priority` (`high` / `medium` / `low`), `reason`, `last_seen` | Follow-up queue |

**Base tables the dashboard also reads:**

| Table | Why |
|-------|-----|
| `automation.lead_log` | Per-contact conversation timeline on conversation detail page |
| `automation.escalations` | Handoffs list (filtered to exclude error records — see data-quality note below) |

### Write surface (new — added by the dashboard)

Schema: `dashboard` (created by this project's migrations in each tenant DB). Isolated from `automation` so the dashboard's tables never interfere with the WhatsApp runtime.

**`dashboard.allowed_emails`**

| Field | Type | Notes |
|-------|------|-------|
| `email` | `text` | Primary key, lowercase |
| `role` | `text` | `viewer` only in v1 (forward-compat) |
| `created_at` | `timestamptz` | Default `now()` |
| `created_by` | `text` | Operator note (e.g. `"jonatan"`) |

**`dashboard.magic_link_tokens`**

| Field | Type | Notes |
|-------|------|-------|
| `token_hash` | `text` | Primary key. SHA-256 of the token sent by email |
| `email` | `text` | References `allowed_emails.email` at auth time |
| `expires_at` | `timestamptz` | Default `now() + 15 minutes` |
| `consumed_at` | `timestamptz` | Null until first use; single-use enforcement |
| `created_at` | `timestamptz` | Default `now()` |

**`dashboard.audit_log`**

| Field | Type | Notes |
|-------|------|-------|
| `id` | `bigserial` | PK |
| `email` | `text` | Who |
| `action` | `text` | `login`, `logout`, `export_csv`, `login_denied` |
| `metadata` | `jsonb` | Context: IP, user-agent, export filters |
| `created_at` | `timestamptz` | Default `now()` |

### Relationships

- `magic_link_tokens.email` logically references `allowed_emails.email` but is not FK-enforced (allowlist can change without invalidating in-flight tokens; the login flow re-checks allowlist membership).
- `audit_log` is append-only, unindexed on `email` (low cardinality per tenant).

### Database Schema (Drizzle)

### Raw SQL migration (`migrations/0000_init.sql`)

```sql
CREATE SCHEMA IF NOT EXISTS dashboard;

CREATE TABLE IF NOT EXISTS dashboard.allowed_emails (
  email       text PRIMARY KEY,
  role        text NOT NULL DEFAULT 'viewer',
  created_at  timestamptz NOT NULL DEFAULT now(),
  created_by  text
);

CREATE TABLE IF NOT EXISTS dashboard.magic_link_tokens (
  token_hash   text PRIMARY KEY,
  email        text NOT NULL,
  expires_at   timestamptz NOT NULL,
  consumed_at  timestamptz,
  created_at   timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_magic_link_tokens_email
  ON dashboard.magic_link_tokens (email);
CREATE INDEX IF NOT EXISTS idx_magic_link_tokens_expires_at
  ON dashboard.magic_link_tokens (expires_at);

CREATE TABLE IF NOT EXISTS dashboard.audit_log (
  id          bigserial PRIMARY KEY,
  email       text,
  action      text NOT NULL,
  metadata    jsonb,
  created_at  timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_audit_log_created_at
  ON dashboard.audit_log (created_at DESC);

-- Read-only role for the dashboard app
DO $$
BEGIN
  IF NOT EXISTS (SELECT 1 FROM pg_roles WHERE rolname = 'dashboard_app') THEN
    CREATE ROLE dashboard_app LOGIN PASSWORD :'dashboard_app_password';
  END IF;
END $$;

GRANT USAGE ON SCHEMA automation TO dashboard_app;
GRANT SELECT ON ALL TABLES IN SCHEMA automation TO dashboard_app;
ALTER DEFAULT PRIVILEGES IN SCHEMA automation GRANT SELECT ON TABLES TO dashboard_app;

GRANT USAGE, CREATE ON SCHEMA dashboard TO dashboard_app;
GRANT ALL ON ALL TABLES IN SCHEMA dashboard TO dashboard_app;
GRANT ALL ON ALL SEQUENCES IN SCHEMA dashboard TO dashboard_app;
```

### Data-quality fix bundled with v1

The `whatsapp-automation-claude` README flags that `automation.escalations` is polluted with error records from `v2-error-handler.json`. The dashboard v1 rollout includes this **upstream** fix (applied to `postgres-setup.sql` + the error handler workflow):

```sql
ALTER TABLE automation.escalations
  ADD COLUMN IF NOT EXISTS escalation_type text NOT NULL DEFAULT 'business';

UPDATE automation.escalations
  SET escalation_type = 'error'
  WHERE reason ILIKE '%error%';
```

`v2-error-handler.json` writes `escalation_type = 'error'` going forward. `v_handoff_summary` and all dashboard queries filter `WHERE escalation_type = 'business'`. This keeps the dashboard clean without brittle string matching.

## 5. API Design

The dashboard is a mostly-no-API app thanks to React Server Components — pages query Postgres directly. The only HTTP routes exposed:

### Routes Overview

| Method | Path | Description | Auth |
|--------|------|-------------|------|
| `GET` | `/api/auth/[...nextauth]` | Auth.js endpoints (signin, callback, signout, session) | Public |
| `GET` | `/api/export/conversations` | Streamed CSV of filtered contacts | Session required |
| `GET` | `/api/export/daily-metrics` | Streamed CSV of daily metrics | Session required |

That's it. Every data page uses Server Components + Drizzle directly — no REST layer to maintain.

### Key Endpoint Detail

**`GET /api/export/conversations`**

Query params:
- `from` — ISO date (optional, defaults to 30 days ago)
- `to` — ISO date (optional, defaults to today)
- `search` — substring against `display_name` or `contact_wa_id` (optional)

Response: `text/csv; charset=utf-8`, streamed row-by-row. Headers: `Content-Disposition: attachment; filename="conversations-{from}-{to}.csv"`.

CSV columns: `contact_wa_id`, `display_name`, `first_seen`, `last_seen`, `message_count`, `last_intent`, `handoff_count`. Dates formatted as DD/MM/YYYY HH:mm (BA timezone).

Auth: rejects with 401 if no valid session. Logs to `dashboard.audit_log` on success.

**`GET /api/export/daily-metrics`**

Query params:
- `from`, `to` — same as above

CSV columns: `day`, `inbound_count`, `outbound_count`, `unique_contacts`, `handoff_count`, `handoff_rate`. Rows ordered by day ascending.

### Rate limiting

Simple in-process token bucket on both export endpoints: max 10 requests per session per minute. Implemented in `src/lib/rate-limit.ts`. Not worth Redis for this scale.

### State Management

- **Server-first.** Pages are Server Components that `await` Drizzle queries. Render time = query time (~50 ms on local Postgres).
- **No global client state.** No Zustand, no Redux, no React Query on the data path.
- **Client state is local.** TanStack Table's pagination/sort/filter live in URL search params (`useSearchParams`) so state is shareable and back-button-friendly.
- **Revalidation:** a "Refresh" button in the header calls `router.refresh()` — that invalidates the Server Component cache and re-runs queries. No realtime / websockets in v1.
- **Loading UX:** `loading.tsx` files provide suspense fallbacks per route segment.

### Rendering model summary

| Component type | Runs where | Examples |
|----------------|-----------|----------|
| Server Components (default) | VPS Node process | Pages, `TenantBadge`, layout data loading |
| Client Components (`"use client"`) | Browser | Charts (Recharts), tables (TanStack), sidebar, user menu, forms |

## 7. Design System

### Philosophy

The visual system lifts directly from `whatsapp-automation-claude/scripts/build-dashboard.py` which already defined a coherent operator-facing UX spec. White-label knobs (logo, primary color) overlay on top of a neutral, professional base so the dashboard feels like *the client's* tool.

### Colors

**Base tokens** (Tailwind CSS v4 via `@theme` in `globals.css`):

| Role | Hex | Usage |
|------|-----|-------|
| Primary | `var(--client-primary)` — default `#3b82f6` | Active nav, primary buttons, chart primary series |
| Background | `#fafafa` (light) / `#0b1220` (dark optional) | Page background |
| Surface | `#ffffff` / `#111827` | Cards, panels |
| Text | `#111827` / `#f3f4f6` | Body text |
| Muted | `#6b7280` | Secondary text, axis labels |
| Border | `#e5e7eb` / `#1f2937` | Card borders, dividers |
| Destructive | `#dc2626` | Errors, delete confirmations |
| Success | `#059669` | Up-trend deltas, successful exports |
| Warning | `#d97706` | Down-trend deltas in positive metrics |

**Priority color tokens** (from Python prototype spec, lifted verbatim):

| Priority | Hex | Usage |
|----------|-----|-------|
| High | `#F4CCCC` (bg) + `#8A1A1A` (text) | Follow-up queue rows |
| Medium | `#FCE5CD` (bg) + `#8A4B00` (text) | Follow-up queue rows |
| Low | `#E6F4EA` (bg) + `#1B5E20` (text) | Follow-up queue rows |

### Typography

| Role | Font | Size | Weight |
|------|------|------|--------|
| Display (KPI values) | Geist Sans | 36px | 600 |
| Headings (h1–h3) | Geist Sans | 28 / 20 / 16 px | 600 |
| Body | Geist Sans | 14px | 400 |
| Table cells | Geist Sans | 13px | 400 |
| Code / numbers (monospace emphasis) | Geist Mono | 14px | 500 |

Geist is already in the BotArgento landing page stack — consistent identity across the ecosystem.

### Spacing & Layout

- Spacing scale: Tailwind default (`4 8 12 16 24 32 48 64`)
- Border radius: `6px` default (`rounded-md`), `8px` for cards, `9999px` for pills/avatars
- Max content width: `1280px`
- Sidebar width: `240px` expanded / `64px` collapsed
- Header height: `56px`
- Card padding: `24px`
- Breakpoints: Tailwind default (`sm 640 / md 768 / lg 1024 / xl 1280`)

### Component style

- **Flat surfaces with subtle borders** (`border-border` + 1px), minimal shadows. Reserved shadow (`shadow-sm`) for hover states only.
- **Dense but breathable.** Internal tools should be information-dense; avoid corporate-site whitespace.
- **No gratuitous animation.** Transitions limited to 150ms on color/opacity. No slide-in on page load.
- **Number formatting:** thousands separators by locale (`1.234` in es-AR), percentages to one decimal, dates `DD/MM/YYYY`.
- **Charts:** gridlines muted (`#f3f4f6`), axis labels `12px` in muted color, tooltips use surface color with 1px border.

### Locale

- Timezone: `America/Argentina/Buenos_Aires` (overridable via `CLIENT_TIMEZONE`)
- Locale: `es-AR` (overridable via `CLIENT_LOCALE`)
- Date display: `DD/MM/YYYY`
- Number display: `1.234,56`
- All Spanish copy lives in `src/config/verticals/<vertical>.ts`; no hardcoded UI strings in components

### Reserved Operations addendum (post-v1)

The "Reserved Operations" redesign supersedes the original flat-Geist spec
above without removing it — older components still in the codebase reference
the original tokens, and the addendum below describes the *current* canonical
direction for new work. Treat the older table as historical; see
[`CLAUDE.md`](../CLAUDE.md#design-system) for the live token list.

- **Display font: Fraunces** (variable axes `SOFT` + `opsz`, exposed as
  `--font-fraunces`). Used for page mastheads, section headings, and hero
  KPI values. Geist Sans stays the body face; Geist Mono stays for numerics
  + kicker captions.
- **Accent-only color**: monochrome canvas with the tenant's
  `--client-primary` as the only colored element. Cards carry a 2px top
  border in the accent (the "masthead rule").
- **Editorial rhythm**: every top-level page section has a mono kicker
  caption + Fraunces section heading. The overview is structured as
  Volumen diario → Intenciones → Operativo → Seguimiento.
- **Theming control plane**: `--client-primary` is no longer just an env
  var. See "Theming control plane" below for the storage + UI + audit flow.

### Theming control plane

|---|---|
| **Storage** | `dashboard.app_settings` — single-row table per tenant DB. PK constrained to `id = 1`. Migration `0002_app_settings.sql` creates + back-fills from `CLIENT_PRIMARY_COLOR`. |
| **Read** | `getAppSettings()` in `src/lib/queries/app-settings.ts`. Returns the persisted row, falls back to env defensively (post-migration the row always exists). |
| **Write** | `updateAppSettings({primaryColor}, by)` — same module. Validates hex, upserts on `id=1`, audits via `dashboard.audit_log` with `action='theme_update'` and `metadata={from, to}`. |
| **API** | `POST /api/settings/theme` — Zod-validated, role-guarded (`requireRole("admin")`). |
| **UI** | `/settings` page (Server Component, `requireRole("admin")` first) renders `<ColorPicker>` (Client Component). The picker writes to `document.documentElement.style` for live preview, then POSTs on save and `router.refresh()`s. |
| **Layout injection** | `src/app/layout.tsx` awaits `getAppSettings()` and injects `--client-primary` inline on `<body>` so the var is set before any paint. |

The pattern for a new admin-controlled setting is:
1. Add a column to `dashboard.app_settings` in a fresh additive migration.
2. Extend `AppSettings` type + getter/setter in `src/lib/queries/app-settings.ts`.
3. Add the field to the `/settings` page (still guarded by
   `requireRole("admin")`); reuse `<ColorPicker>` as the model for live-preview
   inputs.
4. Audit the change via `auditLog` insert with a stable `action` string.

## 8. Authentication & Authorization

### Auth Flow (magic link)

```
1. User visits dashboard.clientN.botargento.com.ar → middleware redirects to /login
2. User enters email → POST form action
3. Server:
     a. Lowercase email
     b. SELECT 1 FROM dashboard.allowed_emails WHERE email = $1
        → if not found: log `login_denied`, return generic success message
           (do NOT reveal allowlist membership)
     c. Generate cryptographically random token (32 bytes, hex)
     d. Store SHA-256(token) in dashboard.magic_link_tokens, expires in 15 min
     e. Send email via Resend with link:
           https://dashboard.clientN.botargento.com.ar/verify?token=<token>
4. User clicks email link → /verify page
5. Server:
     a. SHA-256(token) → lookup in dashboard.magic_link_tokens
     b. Check expires_at > now AND consumed_at IS NULL
     c. Re-check email is in dashboard.allowed_emails (handles off-boarding during in-flight token)
     d. UPDATE consumed_at = now()
     e. Auth.js issues signed JWT cookie (7-day lifetime)
     f. Log `login` to dashboard.audit_log
     g. Redirect to /
6. User is authenticated; middleware checks JWT on every (dashboard)/* request
```

### Protected Routes

- Public: `/login`, `/verify`, `/api/auth/*`, static assets
- Protected (via `middleware.ts`): everything under `(dashboard)/`, `/api/export/*`

### Roles & Permissions

The `role` column in `dashboard.allowed_emails` is enforced server-side via
`src/lib/role-guard.ts`. Pages and route handlers that require elevation call
`await requireRole("admin")` at the top; the guard redirects viewers and
emits a `role_denied` audit row.

| Role | Can Do |
|------|--------|
| `viewer` | View all dashboard pages, export CSV |
| `admin` | Everything `viewer` can do, plus change tenant-scoped settings (currently `--client-primary` via `/settings`) |

The first admin is bootstrapped during `scripts/provision-tenant.sh` (a
prompt collects an email and seeds it as `admin`). Existing tenants promote
via `UPDATE dashboard.allowed_emails SET role='admin' WHERE email='…';`.

### Session Management

- JWT stored in `__Secure-next-auth.session-token` cookie (HttpOnly, Secure, SameSite=Lax)
- Lifetime: 7 days rolling
- Signing secret: `AUTH_SECRET` env var (32-byte hex, rotate periodically per tenant)
- No server-side session table — Auth.js default with `session: { strategy: "jwt" }`

### Security checklist

- Magic link tokens are single-use (enforced via `consumed_at`)
- Tokens are stored hashed (SHA-256), never in plaintext
- 15-minute expiry prevents stale-link abuse
- Allowlist is re-checked at verify time — removing an email invalidates in-flight tokens
- CORS: locked to the tenant subdomain (no cross-origin requests expected)
- All cookies `HttpOnly` + `Secure` — never readable by client JS
- Rate limit on `/login` email submission (5 per IP per hour) to prevent email flooding
- Every auth-relevant action (login, denied, logout, CSV export) written to `audit_log`

### Step 4: Auth.js v5 + Resend magic link + allowlist

Create `src/lib/auth.ts`:

```ts
import NextAuth from "next-auth";
import Resend from "next-auth/providers/resend";
import { db } from "@/db/client";
import { allowedEmails, magicLinkTokens, auditLog } from "@/db/schema";
import { eq, and, gt, isNull } from "drizzle-orm";
import { createHash, randomBytes } from "node:crypto";

export const { handlers, auth, signIn, signOut } = NextAuth({
  session: { strategy: "jwt", maxAge: 60 * 60 * 24 * 7 },
  pages: { signIn: "/login", verifyRequest: "/login?sent=1" },
  providers: [
    Resend({
      from: process.env.AUTH_EMAIL_FROM,
      async sendVerificationRequest({ identifier: email, url, provider }) {
        const token = new URL(url).searchParams.get("token");
        if (!token) throw new Error("no token");
        const hash = createHash("sha256").update(token).digest("hex");

        // Allowlist check — do NOT send if email not allowed
        const allowed = await db.select().from(allowedEmails)
          .where(eq(allowedEmails.email, email.toLowerCase())).limit(1);
        if (allowed.length === 0) {
          await db.insert(auditLog).values({
            email, action: "login_denied", metadata: { reason: "not_in_allowlist" }
          });
          return; // silent — don't reveal membership
        }

        await db.insert(magicLinkTokens).values({
          tokenHash: hash,
          email: email.toLowerCase(),
          expiresAt: new Date(Date.now() + 15 * 60 * 1000),
        });

        // Send via Resend
        await fetch("https://api.resend.com/emails", {
          method: "POST",
          headers: {
            "Authorization": `Bearer ${provider.apiKey}`,
            "Content-Type": "application/json",
          },
          body: JSON.stringify({
            from: provider.from,
            to: email,
            subject: `Acceso a ${process.env.CLIENT_NAME}`,
            html: renderMagicLinkEmail(url),
          }),
        });
      },
    }),
  ],
  callbacks: {
    async signIn({ user }) {
      const email = user.email?.toLowerCase();
      if (!email) return false;
      const allowed = await db.select().from(allowedEmails)
        .where(eq(allowedEmails.email, email)).limit(1);
      if (allowed.length === 0) return false;
      await db.insert(auditLog).values({ email, action: "login" });
      return true;
    },
  },
});
```

## 12. Deployment Strategy

This section is the operational spine. It covers: one-time VPS setup, first-time per-client deploy ceremony, CI/CD pipeline, routine updates, and rollback.

### 12.1 Hosting model

- **Target:** the existing Hostinger VPS that already hosts all n8n tenants.
- **Per tenant:** one Docker container named `dashboard-<clientN>` running alongside that client's `n8n-<clientN>` container and `postgres-<clientN>` container, all on the same Docker network.
- **Routing:** Traefik (already running on the VPS) routes `dashboard.<clientN>.botargento.com.ar` to the container via Docker labels. TLS auto-provisioned via Let's Encrypt.
- **Isolation:** each tenant's dashboard only has credentials for that tenant's DB. A compromise of one tenant's dashboard cannot reach another tenant's data.

### 12.2 One-time VPS preparation (done once, ever)

Prerequisites on the VPS:
1. Docker 24+ with Compose v2 installed
2. Traefik v3 already running (it is — serving n8n tenants)
3. DNS delegated to the right nameservers (Donweb, already configured)
4. A `deploy` system user with membership in the `docker` group
5. SSH key-based access for GitHub Actions → `deploy@vps`

**Tenant directory convention (existing):** per-tenant assets already live at `/opt/n8n/<clientN>/` — this directory hosts the n8n + Postgres compose file, `.env`, and any related volumes. The dashboard adds files *into the same directory* rather than creating a new tree. The `n8n` in the path name is historical; don't rename it.

**Traefik / network conventions** (extracted from the current `/opt/n8n/client1/docker-compose.yml` — keep these identical for the dashboard):

| Item | Value |
|------|-------|
| Public Traefik network | `traefik-public` (declared `external: true, name: traefik-public` in n8n compose) |
| Tenant-internal network | `client1-internal` (declared inside the n8n compose, `internal: true` — Postgres only reachable here) |
| Postgres service name | `postgres` (reachable as `postgres:5432` from any container on `client1-internal`) |
| Postgres container name | `n8n-client1-postgres` |
| n8n container name | `n8n-client1` |
| Entrypoint | `websecure` |
| Cert resolver | `letsencrypt` |
| Container naming convention | `n8n-<clientN>-<service>` (so dashboard = `n8n-<clientN>-dashboard`) |
| Traefik router naming convention | `<service>-<clientN>` for uniqueness (n8n uses `client1`; dashboard uses `dashboard-client1`) |
| Timezone env | `${TZ}` sourced from `.env` — mirrored into `dashboard.env` |

**Critical network gotcha** — `client1-internal` is created by the n8n compose file without `external: true`, so Docker prefixes it with the compose project name. Before launching the dashboard, run:

```bash
docker network ls | grep client1
# Expect something like:
#   abc123...   client1_client1-internal   bridge   local
#   def456...   traefik-public             bridge   local
```

Take the **exact name shown** (e.g. `client1_client1-internal`) and use it in the dashboard compose under `networks.<name>.name`. This is the sole cross-compose-file coupling.

### 12.3 First-time deployment for a new client (the ceremony)

**This is the runbook for every new client. Target time: ≤ 15 minutes.**

The client's n8n runtime and Postgres must already exist — dashboard provisioning depends on them.

#### Step A — Prerequisites checklist

Before starting, confirm:
- [ ] Client's n8n is running at `https://<clientN>.botargento.com.ar`
- [ ] Client's Postgres is running and contains `automation` schema with all 5 reporting views
- [ ] Client has provided 1-5 emails that need dashboard access
- [ ] Client's brand assets available: logo SVG, primary hex color, legal name

#### Step B — DNS record

In Donweb DNS for `botargento.com.ar`:

```
Type:  CNAME
Name:  dashboard.<clientN>
Value: <vps-host>.botargento.com.ar   (or the VPS public hostname)

```

Propagation typically takes 2-5 minutes. Verify with `dig dashboard.<clientN>.botargento.com.ar` before proceeding.

#### Step C — Prepare the tenant DB

On the VPS, as the `deploy` user, with a psql shell into the client's Postgres. The client's existing tenant directory is `/opt/n8n/<clientN>/`. Add a `migrations/` subdirectory there for dashboard-specific SQL:

```bash
mkdir -p /opt/n8n/<clientN>/migrations
# Copy the migration SQL files from the dashboard repo or fetch them from GitHub raw:
curl -fsSL https://raw.githubusercontent.com/jperez1804/botargento-dashboard/main/migrations/0000_init.sql \
  -o /opt/n8n/<clientN>/migrations/0000_init.sql
curl -fsSL https://raw.githubusercontent.com/jperez1804/botargento-dashboard/main/migrations/0001_escalation_type.sql \
  -o /opt/n8n/<clientN>/migrations/0001_escalation_type.sql

# Generate a strong password for the dashboard's DB user
DASHBOARD_APP_PASSWORD=$(openssl rand -hex 24)
echo "$DASHBOARD_APP_PASSWORD"   # Save this — it goes into the env file

# Source the tenant env file to get POSTGRES_USER, POSTGRES_PASSWORD, POSTGRES_DB
set -a; source /opt/n8n/<clientN>/.env; set +a

# Apply via the running postgres container (no network exposure needed)
docker exec -i n8n-<clientN>-postgres \
  psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" \
       -v dashboard_app_password="'$DASHBOARD_APP_PASSWORD'" \
  < /opt/n8n/<clientN>/migrations/0000_init.sql

# If the data-quality fix isn't already applied to automation.escalations, apply it now:
docker exec -i n8n-<clientN>-postgres \
  psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" \
  < /opt/n8n/<clientN>/migrations/0001_escalation_type.sql
```

We run migrations **via `docker exec`** because `client1-internal` is marked `internal: true` (no external network access to Postgres by design). Connecting via the container avoids exposing a Postgres port on the host.

The migration creates:
- `dashboard` schema with `allowed_emails`, `magic_link_tokens`, `audit_log`
- Role `dashboard_app` with read access to `automation.*` and read-write on `dashboard.*`

#### Step F — Create the docker-compose fragment

The dashboard lives **in the same directory** as the existing n8n setup (`/opt/n8n/<clientN>/`), as a second compose file. This keeps one tenant = one directory. Do **not** modify the existing `docker-compose.yml` that runs n8n + Postgres — add a new file instead.

Create `/opt/n8n/<clientN>/dashboard.compose.yml` (shown with `<clientN>` = `client1` filled in — duplicate and edit for each new tenant):

```yaml
services:
  dashboard:
    image: ghcr.io/jperez1804/dashboard:${DASHBOARD_TAG:-latest}
    container_name: n8n-client1-dashboard
    restart: unless-stopped
    expose:
      - "3000"
    environment:
      # Postgres: reachable as hostname `postgres` via the tenant-internal network.
      # Credentials: `dashboard_app` was created by migrations/0000_init.sql
      TENANT_DB_URL: "postgres://dashboard_app:${DASHBOARD_APP_PASSWORD}@postgres:5432/${POSTGRES_DB}"

      VERTICAL: real-estate
      CLIENT_NAME: "Inmobiliaria Pepito"
      CLIENT_LOGO_URL: "/logos/client.svg"
      CLIENT_PRIMARY_COLOR: "#4a7ec4"
      CLIENT_TIMEZONE: ${TZ}
      CLIENT_LOCALE: "es-AR"

      AUTH_SECRET: "${AUTH_SECRET}"
      AUTH_URL: "https://dashboard.client1.botargento.com.ar"
      AUTH_EMAIL_FROM: "no-reply@botargento.com.ar"
      RESEND_API_KEY: "${RESEND_API_KEY}"

      TZ: ${TZ}
      NODE_ENV: production
      LOG_LEVEL: info
    volumes:
      - /opt/n8n/client1/assets/logo.svg:/app/public/logos/client.svg:ro
    networks:
      - client1-internal       # reach postgres by service name
      - traefik-public         # accept ingress via Traefik
    labels:
      - "traefik.enable=true"
      - "traefik.docker.network=traefik-public"
      - "traefik.http.routers.dashboard-client1.rule=Host(`dashboard.client1.botargento.com.ar`)"
      - "traefik.http.routers.dashboard-client1.entrypoints=websecure"
      - "traefik.http.routers.dashboard-client1.tls=true"
      - "traefik.http.routers.dashboard-client1.tls.certresolver=letsencrypt"
      - "traefik.http.routers.dashboard-client1.service=dashboard-client1"
      - "traefik.http.services.dashboard-client1.loadbalancer.server.port=3000"

networks:
  # 'name:' must match the actual Docker network name on the VPS.
  # For this VPS's convention (verified 2026-04-23):
  #   docker network ls | grep <clientN>
  #   → <clientN>_<clientN>-internal
  # This is because Compose prefixes the network with the project name,
  # which defaults to the directory name (/opt/n8n/<clientN> → project = <clientN>).
  # Pattern for every tenant: name: <clientN>_<clientN>-internal
  client1-internal:
    external: true
    name: client1_client1-internal   # ← for client2: client2_client2-internal

  traefik-public:
    external: true
    name: traefik-public
```

### Why the networking looks like this

- **`client1-internal` (internal: true)** — created by the n8n compose. Postgres is on it. No host ports, no internet. Declaring `external: true` on our side means "don't create it, just attach." The `name:` field pins the exact Docker network name (because the n8n compose didn't pin it, so it got a project-prefixed name).
- **`traefik-public`** — already `external: true, name: traefik-public` in n8n compose. We attach the same way. This is how Traefik reaches the dashboard AND how the dashboard reaches the public internet (for Resend API calls).
- **No `depends_on: postgres`** — that's a cross-compose-file concern Docker can't express. The container entrypoint runs `wait-for-postgres` before starting Next.js, so a cold boot while postgres is restarting just waits gracefully.
- **Container name follows your `n8n-clientN-*` convention** so `docker ps` groups tenants visually.
- **Traefik router name must be unique per router**: n8n already uses `client1`, so dashboard uses `dashboard-client1`. The explicit `service=dashboard-client1` label prevents Traefik auto-naming collisions.

Create the matching `.env` file at `/opt/n8n/<clientN>/dashboard.env` (file mode `0600`). It **reuses** the same `POSTGRES_DB` and `TZ` values from the tenant's existing `/opt/n8n/<clientN>/.env` — keep them in sync:

```bash
# Reused from the existing .env (must match)
POSTGRES_DB=<same value as in /opt/n8n/clientN/.env>
TZ=America/Argentina/Buenos_Aires

# New, dashboard-only
DASHBOARD_APP_PASSWORD=<value from Step C>
AUTH_SECRET=<openssl rand -hex 32>
RESEND_API_KEY=<from Resend dashboard>
DASHBOARD_TAG=latest
```

```bash
chmod 0600 /opt/n8n/<clientN>/dashboard.env
```

**Why a separate env file?** Keeping n8n's `.env` untouched means dashboard deploys never risk breaking n8n. The two files pass to two different `docker compose` invocations. The small duplication (`POSTGRES_DB`, `TZ`) is worth the isolation.


### 12.5 `update-dashboards.sh` — the VPS deploy script

Lives at `/opt/scripts/update-dashboards.sh`. Read by the `deploy.yml` workflow, but also runnable manually.

```bash
#!/usr/bin/env bash
# /opt/scripts/update-dashboards.sh
set -euo pipefail

TENANT_FILTER="${1:-all}"
TAG="${2:-latest}"
TENANT_ROOT="/opt/n8n"
REGISTRY="/opt/scripts/tenants.txt"

if [[ "$TENANT_FILTER" == "all" ]]; then
  # Source of truth for which tenants have a dashboard deployed
  TENANTS=$(cat "$REGISTRY")
else
  TENANTS="$TENANT_FILTER"
fi

for t in $TENANTS; do
  COMPOSE_FILE="$TENANT_ROOT/$t/dashboard.compose.yml"
  ENV_FILE="$TENANT_ROOT/$t/dashboard.env"

  if [[ ! -f "$COMPOSE_FILE" ]]; then
    echo "⚠  $t has no dashboard.compose.yml — skipping"
    continue
  fi

  CONTAINER="n8n-$t-dashboard"
  echo "→ Updating $CONTAINER to tag '$TAG'"

  # Pin the tag in the env file so restarts use the same version
  sed -i "s/^DASHBOARD_TAG=.*/DASHBOARD_TAG=$TAG/" "$ENV_FILE"

  cd "$TENANT_ROOT/$t"
  # Service name in the compose file is 'dashboard' (not per-tenant),
  # container_name is what's per-tenant.
  docker compose --env-file dashboard.env -f dashboard.compose.yml pull dashboard
  docker compose --env-file dashboard.env -f dashboard.compose.yml up -d dashboard

  # Wait for health (30s max)
  for i in $(seq 1 30); do
    if docker inspect "$CONTAINER" --format '{{.State.Status}}' 2>/dev/null | grep -q "running"; then
      echo "✓ $CONTAINER is running"
      break
    fi
    sleep 1
  done
done

echo "✓ All done"
```

### 12.6 Rollback

Every image is tagged with its commit SHA. Rollback = deploy the previous-known-good SHA.

1. Find the last-known-good SHA in the **Release** workflow run history
2. Run the **Deploy** workflow with `tag: <that-SHA>` and `tenant: all`
3. ~2 minutes later, all tenants are back on the previous version

No database rollback needed — Drizzle migrations are additive and the app supports reading from both the old and new schema shapes (maintain forward-compatible migrations as a discipline).

### 12.7 Environments

v1 has exactly two:

| Env | How | When to use |
|-----|-----|-------------|
| Local dev | `pnpm dev` + local Postgres | Writing code |
| Production (per tenant) | VPS Docker container | Everything else |

**No staging in v1.** Rationale: the app is read-only from the runtime perspective; regressions are visible (charts wrong, pages broken) but not destructive. Ship to one canary tenant first when you want extra caution — you can adopt this informal rule without building a staging environment.

If/when staging is needed later: stand up a synthetic tenant (`dashboard.staging.botargento.com.ar`) pointing at a copy of a real tenant's DB. Low effort — same provisioning runbook.

## 13. Testing Strategy

### Unit (Vitest)

Located at `tests/unit/**`. Run with `pnpm test`.

- **Query builders** (`src/lib/queries/*.ts`) — mock the Drizzle client, assert the SQL shape via Drizzle's `.toSQL()`.
- **CSV formatter** (`src/lib/csv.ts`) — given sample rows, assert byte-identical CSV output including escaping.
- **Date helpers** (`src/lib/date.ts`) — tz conversion, DD/MM/YYYY formatting.
- **Allowlist check** — pure function, given email + allowlist rows, returns boolean.
- **Tenant config parsing** — Zod schema on bad env vars should throw with a clear message.

Coverage target: meaningful coverage on `src/lib/**`. Don't chase line coverage on components.

### Integration (Vitest + Postgres service)

Run against the CI Postgres service container.

- Migrations apply cleanly on a fresh DB and are idempotent.
- Seeded data produces the expected view row counts.
- A full query path (Drizzle → Postgres → view) returns the expected shape for each page.

### E2E (Playwright)

Located at `tests/e2e/**`. Run with `pnpm test:e2e`.

Minimum suite:
- **Login flow** — POST email → read magic link from DB → GET verify URL → assert dashboard visible.
- **Overview renders** — after login, all 4 KPI cards show numbers, volume chart SVG present, intents chart SVG present, no error boundaries.
- **Conversations list** — search box filters rows, pagination works, clicking a row navigates to detail.
- **CSV export** — clicking Export downloads a file; assert content-type and first line header.
- **Auth guard** — hitting `/` without session redirects to `/login`.

Run E2E only in CI against a dedicated test tenant setup, not on every PR (slow). Gate them on a path filter: any change to `src/app/**` or `src/components/**`.

```markdown
## Commands

## Tech Stack

Next.js 15 (App Router) + TypeScript strict + Tailwind CSS v4 + shadcn/ui + Recharts + TanStack Table + Drizzle ORM + Postgres + Auth.js v5 (Resend magic link) + Docker + Traefik

## Architecture

### Directory Structure
- `src/app/(auth)/` — Public pages: login, verify
- `src/app/(dashboard)/` — Protected pages: overview, conversations, handoffs, follow-up
- `src/app/api/` — Route handlers: auth callback, CSV exports
- `src/components/dashboard/` — Page-specific visual components (KPI cards, charts, tables)
- `src/components/layout/` — Shell components (sidebar, header, branding)
- `src/components/ui/` — shadcn primitives
- `src/config/verticals/` — Swappable vertical configs; v1 ships `real-estate.ts`
- `src/config/tenant.ts` — Runtime config read from CLIENT_* env vars
- `src/db/` — Drizzle schema + client + typed wrappers for `automation.v_*` views
- `src/lib/queries/` — All SQL lives here; pages call these functions, never inline SQL
- `src/lib/auth.ts` — Auth.js config with Resend magic link + allowlist check
- `src/middleware.ts` — Auth guard for `(dashboard)/*` routes
- `migrations/` — Raw SQL migrations for `dashboard.*` schema (applied on container start)
- `scripts/` — Provisioning, seeding, view-compat verification
- `.github/workflows/` — CI, release (Docker image), deploy (SSH to VPS)

### Data Flow
Server Components await Drizzle queries from `src/lib/queries/*.ts` and render HTML on the VPS. Client Components (`"use client"`) only handle interactivity (charts, tables, forms). No REST API for data — pages query Postgres directly.

### Key Patterns
- **Server Components by default.** Only add `"use client"` when a component needs React state, effects, or event handlers.
- **All SQL goes through `src/lib/queries/*.ts`.** Never inline queries inside page components.
- **All UI labels come from `verticalConfig`.** Zero hardcoded Spanish strings in JSX.
- **All tenant branding comes from `tenantConfig`** (parsed from `CLIENT_*` env vars). White-label from day one.
- **Reads are read-only to `automation.*`.** Writes only touch `dashboard.*`. Enforced at the DB-role level via `dashboard_app`.
- **Migrations run on container start** via `scripts/container-entrypoint.sh`. View-compatibility is verified at boot; the container fails fast if required views are missing.
- **Auth tokens are hashed.** `dashboard.magic_link_tokens` stores SHA-256, never plaintext.
- **Every auth-relevant action goes to `dashboard.audit_log`** — logins, denials, CSV exports.

## Code Organization Rules

1. **One component per file.** Max 300 lines. Extract sub-components when larger.
2. **Path alias:** `@/` → `src/`. Always use it — never relative imports across directories.
3. **No barrel exports.** Import directly from the file that defines the export.
4. **Server Components are the default.** `"use client"` must be justified by actual interactivity needs.
5. **Colocate route-specific components** next to their page. Components used on ≥2 pages go in `src/components/dashboard/` or `src/components/layout/`.
6. **All env vars read via Zod-validated config modules** (`src/config/tenant.ts`, `src/config/env.ts`). Never `process.env.X` directly in feature code.
7. **URL state over component state** for table filters, pagination, date ranges. Keep the URL shareable.
```

## 16. Reglas No Negociables (Blueprint-level, for the builder)

1. **Follow the build order in Section 9.** Don't jump ahead to page 5 before page 3's foundations are done.
2. **Ship the vertical abstraction in Step 7, even though only one vertical exists in v1.** Hardcoding labels now = week of refactor later.
3. **The data-quality fix to `automation.escalations` is in scope for v1.** Don't ship the handoffs page without it — error rows will pollute client trust.
4. **White-label from Step 6, not as a v2 addition.** Hardcoding BotArgento branding and retrofitting later is more painful than the 1 hour it costs to do right the first time.
5. **Run `verify-view-compat.ts` at container startup.** Failing fast with a clear error saves hours of "why does the chart show zeros" debugging.
6. **CSV exports must stream, not buffer.** Tenants with 10k+ contacts will OOM a naive implementation.
7. **No real-time WebSockets in v1.** Server Components + manual refresh is the deal.
8. **Auth.js JWT sessions only.** Don't add a sessions table "just in case."
9. **Deploy workflow is manual (`workflow_dispatch`), not auto-on-merge.** You want the Deploy button between merging and clients seeing the change.
10. **Image tags are git SHAs, not just `:latest`.** Rollback is a tag change, not a cherry-pick + rebuild.
11. **Every tenant's Postgres gets its own `dashboard` schema and its own migrations.** One tenant running ahead on a schema version is expected and safe.
12. **The dashboard is read-only in v1.** If a feature request looks like a write action ("mark as contacted", "add note"), push back to v2. v1 ships as a reporting surface, not an operations tool.
